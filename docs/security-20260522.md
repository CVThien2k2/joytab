# Bảo mật

## 1. Mục tiêu tài liệu
- Tổng hợp các cơ chế bảo mật đang áp dụng trong hệ thống theo trạng thái code hiện tại.

## 2. Xác thực và phân quyền
- Cơ chế xác thực: **Google OAuth 2.0** (passport-google-oauth20, scope `email profile`) + **session cookie** dạng opaque token.
- Sau OAuth thành công, BE cấp session token ngẫu nhiên 32 byte, băm SHA-256 lưu `user_sessions.token_hash`; raw token đặt trong cookie HttpOnly `session_id`. DB **không lưu token raw**.
- `SessionGuard` xác thực từng request bảo vệ: đọc cookie `session_id` + `device_id`, băm token, đối chiếu `token_hash`, kiểm tra khớp `device_id`, chưa revoke và chưa hết hạn.
- Mô hình phân quyền: hiện chưa có RBAC/ABAC — mọi endpoint bảo vệ chỉ yêu cầu session hợp lệ (định danh `userId`). Endpoint quản lý phiên chỉ thao tác trên tài nguyên thuộc chính user.
- Thu hồi quyền: revoke session (`is_revoked = true`) khi logout hoặc thu hồi từ xa; lần request kế tiếp bị chặn bằng `AUTH_004`.

## 3. Cơ chế cookie phiên
- `session_id`: HttpOnly, `SameSite=Lax`, `Secure` khi `NODE_ENV=production`, `path=/`, TTL 7 ngày, sliding renew khi còn dưới 1 ngày.
- `device_id`: HttpOnly, cùng thuộc tính, sống 1 năm để giữ định danh thiết bị.
- `COOKIE_DOMAIN` (tùy chọn, vd `.joytab.com`): dùng cookie first-party chia sẻ giữa FE và API subdomain; để trống ở local → cookie host-only.

## 4. Bảo vệ API
- Rate limit (`@nestjs/throttler`): global 60 req/60s; nhóm `/auth` giới hạn 10 req/60s.
- Security headers: `helmet` bật ở `main.ts` (tắt CSP vì backend chỉ trả JSON/redirect).
- CORS: chỉ cho phép origin từ `FRONTEND_ORIGIN`, bật `credentials: true` để gửi cookie.
- Validation: `ValidationPipe` global (`transform: true`, `whitelist: true`) loại field lạ; DTO dùng `class-validator` (vd `SwitchAccountDto.userId` phải là UUID).
- Xử lý lỗi thống nhất qua `HttpExceptionFilter` + `AppException` (mã lỗi chuẩn).

## 5. Quản lý bí mật và khóa
- Secret lưu qua biến môi trường; `api/.env.example` mô tả biến bắt buộc/tùy chọn, không commit `.env` thật.
- API fail-fast khi thiếu biến bắt buộc (`DB_*`, `REDIS_HOST/PORT/DB`, `GOOGLE_CLIENT_ID/SECRET`, `API_URL`) ngay lúc bootstrap.
- Session token sinh bằng `crypto.randomBytes` và chỉ lưu bản băm SHA-256 → lộ DB không suy ra được token.

## 6. Giám sát và log
- `RequestLoggerMiddleware` log request cho mọi route; `AppLogger` ghi log ra terminal và file `api/logs/YYYY-MM-DD.log` qua queue bất đồng bộ.
- Lưu ý: hiện chưa có cơ chế mask dữ liệu nhạy cảm trong log — cần cân nhắc khi mở rộng.

## 7. Checklist hardening
- [ ] Tắt debug/log dư thừa ở production.
- [ ] Bật HTTPS/TLS ở tầng gateway để cookie `Secure` có hiệu lực.
- [ ] Kiểm tra dependency có lỗ hổng định kỳ.
- [ ] Cân nhắc bật lại CSP nếu backend phục vụ nội dung HTML.
- [ ] Bổ sung mask secret/PII trong log.

## 8. Ghi chú
- Biến `JWT_SECRET` và các dependency `jsonwebtoken` hiện **chưa được sử dụng** trong code (design intent cho khả năng dùng JWT sau này). Mô hình phiên hiện tại không dùng JWT.
